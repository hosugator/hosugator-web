# AWS Placement Groups

## 요약 (Summary)
EC2 인스턴스의 물리적 배치 전략을 논리적으로 정의하여 워크로드의 성격에 최적화된 하드웨어 성능과 가용성을 확보하는 설정

## 배경 (Background)
- 탄생 배경: 클라우드 환경에서는 사용자가 인스턴스의 물리적 위치를 제어할 수 없으므로, 특수한 성능 요구사항(초저지연)이나 고가용성 요구사항(동시 장애 방지)을 충족하기 위한 수단이 필요했음
- 핵심 과제: 인스턴스들이 우연히 같은 랙에 몰려 동시에 고장 나거나, 반대로 너무 멀리 떨어져 통신 속도가 느려지는 통제 불능의 상황을 극복해야 함

## 솔루션 (Solution)
- 핵심 설계: AWS 내부 알고리즘을 통해 인스턴스를 물리적 랙 및 하드웨어 수준에서 전략적으로 배치하는 3가지 논리적 그룹(Cluster, Partition, Spread) 제공
- 작동 메커니즘: 인스턴스 시작 시 특정 배치 그룹을 지정하면, AWS는 해당 그룹의 정책에 따라 인스턴스를 서로 밀착시키거나(Cluster), 물리적으로 격리된 랙에 분산(Spread/Partition)하여 배치함

## 변별점 (Distinction)
- 비유: 팀 프로젝트를 할 때 팀원들을 한 방에 모아 작업 효율을 높일지(Cluster), 비상 상황을 대비해 여러 건물에 나눠 배치할지(Spread) 결정하는 조직 구성 전략
- 대안과의 차이: 
	 - [Default Placement]: AWS가 임의로 인스턴스를 배치하여 일반적인 용도에는 적합하나, 극강의 네트워크 성능이나 랙 단위의 장애 격리가 필요한 엔터프라이즈 워크로드에는 대응하기 어려움
