# AWS Placement Group 유형 비교

## 요약 (Summary)
성능 극대화(Cluster), 대규모 장애 격리(Partition), 개별 장애 최소화(Spread)라는 각기 다른 비즈니스 목적에 따라 EC2의 물리적 위치를 결정하는 세 가지 전략

## 배경 (Background)
- 탄생 배경: 클라우드 인프라의 복잡성이 증가함에 따라, 단순히 서버를 생성하는 것을 넘어 서버 간의 통신 속도나 장애 시의 영향 범위를 정교하게 제어해야 할 필요성이 커짐
- 핵심 과제: 성능과 가용성이라는 상충하는 가치(Trade-off) 사이에서 워크로드 특성에 맞는 최적의 배치를 선택해야 함

## 솔루션 (Solution: 유형별 비교)

| 구분 | Cluster (클러스터) | Partition (파티션) | Spread (스프레드) |
| :--- | :--- | :--- | :--- |
| **핵심 목적** | **초저지연 & 고대역폭** | **대규모 장애 격리** | **최대 가용성 (격리)** |
| **배치 방식** | 단일 AZ 내 물리적 근접 배치 | 논리적 파티션(랙 그룹)별 분산 | 각 인스턴스를 서로 다른 랙에 배치 |
| **장애 영향** | 랙 장애 시 그룹 전체 영향 | 파티션 내 인스턴스만 영향 | 특정 랙 장애 시 1개 인스턴스만 영향 |
| **주요 사례** | HPC, 빅데이터 분석, 인메모리 DB | HDFS, HBase, Cassandra (NoSQL) | 개별 크리티컬 인스턴스, 소규모 DB |
| **네트워크** | 최고 속도 (10Gbps+) | 보통 | 보통 |

## 변별점 (Distinction)
- 비유: 
	 - [Cluster]: 팀원들이 한 책상에 모여 협업하는 것 (소통 최강)
	 - [Partition]: 부서별로 다른 층을 사용하여 한 층에 불이 나도 다른 부서는 살리는 것 (집단 생존)
	 - [Spread]: 모든 팀원이 각자 다른 건물을 사용하여 전염병 확산을 막는 것 (개별 생존)
- 선택 가이드: 
	 - 성능이 최우선이라면 **Cluster**, 대규모 클러스터링 소프트웨어를 운영한다면 **Partition**, 절대 죽으면 안 되는 소수의 서버라면 **Spread**를 선택함
