# Comparison: Lightsail vs EC2 vs ECS vs Lambda

## 비교 목적 (Objective)
워크로드의 규모, 관리 편의성, 비용 구조에 따른 최적의 컴퓨팅 리소스 선정

## 요소별 상세 비교 (Feature Matrix)
| 비교 기준 | [[Lightsail]] | [[EC2]] | [[ECS]] | [[Lambda]] |
| :--- | :--- | :--- | :--- | :--- |
| 핵심 본질 | 올인원 가상 서버 (VPS) | 범용 가상 서버 (IaaS) | 컨테이너 오케스트레이션 | 서버리스 함수 (FaaS) |
| 장점 (Pros) | 극도의 단순함, 고정 비용 | 무한한 제어권과 확장성 | 마이크로서비스 관리에 최적 | 관리 0, 자동 확장, 최저 비용 |
| 단점 (Cons) | 세부 설정 불가 | 관리 오버헤드 높음 | 학습 곡선이 높음 | 실행 시간 제한 (15분) |
| 비용/관리 | 고정 월정액 / 낮음 | 종량제(RI 가능) / 높음 | 인스턴스/Fargate 기반 / 중간 | 호출당 과금 / 거의 없음 |

## 선택 기준 및 로직 (Selection Logic)

 - **Lightsail이 적합한 경우**
	- 조건 (IF): 소규모 웹사이트, 개발 환경, 단순한 앱 배포가 목적일 때
	- 이유 (THEN): 인프라 고민 없이 고정된 예산 안에서 빠르게 시작할 수 있음

- **EC2가 적합한 경우**
	- 조건 (IF): 운영체제 수준의 제어가 필요하거나 특정 하드웨어 성능이 중요할 때
	- 이유 (THEN): 거의 모든 워크로드를 수용할 수 있는 가장 강력한 범용성을 가짐

- **ECS가 적합한 경우**
	- 조건 (IF): 도커 컨테이너 기반의 복잡한 앱을 운영하고 관리해야 할 때
	- 이유 (THEN): 여러 컨테이너의 배치와 확장을 자동화하여 운영 효율을 높임

- **Lambda가 적합한 경우**
	- 조건 (IF): 간헐적인 이벤트 처리나 트래픽 변동이 극심한 API 서버일 때
	- 이유 (THEN): 서버를 켜둘 필요가 없어 비용이 가장 저렴하며 관리가 불필요함

## Conclusion
인프라 관리 능력이 부족하거나 예산이 고정된 초기 프로젝트라면 **Lightsail**로 시작하고, 규모가 커지거나 정교한 제어가 필요해지면 **EC2/ECS**로, 관리 자체를 없애고 싶다면 **Lambda**로 전환하는 전략이 유효합니다.
