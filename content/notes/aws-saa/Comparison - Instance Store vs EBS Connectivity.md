# Comparison - Instance Store vs EBS Connectivity

## 비교 목적 (Objective)
클라우드 환경에서 '로컬 스토리지'와 '네트워크 스토리지'의 물리적 연결 차이를 이해하고, 이에 따른 성능과 데이터 보존 특성을 파악하기 위함입니다.

## 요소별 상세 비교 (Feature Matrix)

| 비교 기준 | [[Amazon EC2 Instance Store]] | [[Amazon EBS]] |
| :--- | :--- | :--- |
| **물리적 위치** | 해당 EC2 인스턴스가 실행 중인 **물리적 서버 내부** | 네트워크로 연결된 별도의 스토리지 서버 랙 |
| **연결 방식** | **물리적 로컬 버스(NVMe/SATA)** 직접 연결 | 이더넷/광케이블 기반의 **전용 네트워크** 연결 |
| **속도(Latency)** | 극도로 낮음 (마이크로초 단위) | 상대적으로 높음 (밀리초 단위) |
| **클라우드 개념** | '물리 장비의 부품'을 임대하는 개념 | '네트워크 드라이브'를 할당받는 개념 |

## 선택 기준 및 로직 (Selection Criteria)

### [[Amazon EC2 Instance Store]]를 선택해야 하는 경우
* **조건:** 데이터가 네트워크 구간을 거치지 않고 메인보드에 직접 꽂힌 SSD 수준의 속도로 처리되어야 하는 경우
    * *Ex:* 물리적인 디스크 쓰기 속도가 곧 서비스 성능인 대규모 NoSQL DB의 캐시 계층

### [[Amazon EBS]]를 선택해야 하는 경우
* **조건:** 저장소가 인스턴스와 물리적으로 떨어져 있더라도, 인스턴스가 고장 났을 때 다른 서버에 다시 연결해서 데이터를 살려야 하는 경우
    * *Ex:* 일반적인 웹 서버의 운영체제(OS) 영역이나 영구 보관이 필요한 사용자 데이터

---
**Conclusion:**
Instance store는 클라우드 가상화 기술을 통해 **'물리 서버에 붙어 있는 실제 하드디스크'**의 제어권을 사용자에게 직접 넘겨주는 방식이기 때문에 로컬 연결이라고 부릅니다.
