# Comparison: RDS vs Athena vs Redshift

## 비교 목적 (Objective)
데이터의 규모, 저장 방식, 그리고 분석 목적에 따라 SQL을 실행할 최적의 관리형 서비스 선정

## 요소별 상세 비교 (Feature Matrix)
| 비교 기준 | [[Amazon RDS]] | [[Amazon Athena]] | [[Amazon Redshift]] |
| :--- | :--- | :--- | :--- |
| 핵심 본질 | 관계형 데이터베이스 (OLTP) | 서버리스 대화형 쿼리 서비스 | 데이터 웨어하우스 (OLAP) |
| 데이터 위치 | DB 인스턴스 내부 전용 스토리지 | S3 버킷 (외부 데이터) | 전용 고성능 클러스터 스토리지 |
| 장점 (Pros) | 트랜잭션 보장, 빠른 읽기/쓰기 | 설정 0, 쓴 만큼만 지불, 서버리스 | 대규모 병렬 처리, 복잡한 통계 분석 |
| 단점 (Cons) | 인스턴스 관리 필요, 스케일링 제한 | 쿼리 성능이 S3 상태에 의존함 | 고비용, 지속적인 운영 관리 필요 |
| 비용/관리 | 시간당 고정 비용 / 중간 | 쿼리당 스캔 용량 과금 / 매우 낮음 | 노드당 고정 비용 / 높음 |

## 선택 기준 및 로직 (Selection Logic)

- Amazon RDS가 적합한 경우
	- 조건 (IF): 애플리케이션의 실시간 데이터 수정 및 조회가 핵심인 서비스일 때
	- 이유 (THEN): 인덱싱과 트랜잭션 무결성을 통해 안정적인 데이터 처리가 가능함

- Amazon Athena가 적합한 경우
	- 조건 (IF): S3에 쌓인 로그나 원본 데이터를 가끔 SQL로 확인만 하면 될 때
	- 이유 (THEN): 데이터를 옮길 필요가 없고, 쿼리할 때만 돈을 내면 되므로 가장 경제적임

- Amazon Redshift가 적합한 경우
	- 조건 (IF): 비즈니스 인텔리전스(BI) 등 대규모 데이터셋에 대한 복잡한 연산이 매일 필요할 때
	- 이유 (THEN): 컬럼형 저장 방식과 병렬 처리를 통해 수십억 행의 데이터도 빠르게 집계 가능함

## Conclusion
가장 가벼운 SQL 실행은 **Athena**, 표준적인 앱 데이터 관리는 **RDS**, 빅데이터 수준의 전문 통계 분석은 **Redshift**를 선택하는 것이 아키텍처의 정석입니다.
