# Compute Workload Options Comparison (컴퓨팅 워크로드 옵션 비교)
## 정의
클라우드에서 애플리케이션을 실행하기 위한 세 가지 근본적인 컴퓨팅 모델(EC2, 컨테이너, 서버리스)의 특성, 장단점, 적합한 사용 사례를 비교하여 워크로드 요구 사항에 맞는 최적의 서비스를 선택하는 기준
## 비교
| 구분 | EC2 (Virtual Machine) | Container (ECS/EKS) | Lambda (Serverless) |
| :--- | :--- | :--- | :--- |
| 추상화 단위 | 하드웨어 (가상 머신) | OS 커널 (컨테이너 이미지) | 코드/함수 (Function as a Service) |
| 운영 책임 | 높음 (OS, 런타임, 확장, 보안 모두 관리) | 중간 (컨테이너 내부만 관리) | 낮음 (코드 실행 환경만 관리) |
| 자유도 | 최고 (OS 커널 구성 변경 등 가능) | 중간 (표준화된 환경 사용) | 최소 (AWS에 의해 제약됨) |
| 가상화 방식 | 하드웨어 가상화 (전체 Guest OS 사용) | OS 커널 공유 (Host OS 커널 사용) | AWS의 자체 컴퓨팅 환경 (사용자에게 완전히 숨겨짐) |
| 시작 시간/오버헤드 | 느림 (분 단위) / 높음 (OS 부팅 필요) | 빠름 (초 단위) / 낮음 (OS 부팅 불필요) | 매우 빠름 (밀리초) / 콜드 스타트 문제 존재 |
| 확장성 | ASG를 통한 수평적 확장 (느림) | ASG/Fargate를 통한 수평적 확장 (빠름) | 자동 확장 (실시간, 무제한) |
| 비용 모델 | 시간당 요금 (유휴 비용 발생) | EC2 시간당 또는 Fargate 사용량 기반 | 요청 및 실행 시간(ms)당 요금 (유휴 비용 없음) |
| 최대 실행 시간 | 무제한 (인스턴스가 실행되는 동안) | 무제한 (컨테이너 태스크 지속 시간 동안) | 15분으로 제한된다. |
## 상세 특징 및 활용
### EC2 (높은 제어권의 필요성)
- 적합 상황: OS 커널 레벨의 커스터마이징이 필요하거나, 특정 하드웨어에 대한 직접적인 접근이 필요한 워크로드이다.
- 예시: 라이선스 제약이 있는 레거시 애플리케이션, HPC 워크로드, 하이퍼바이저를 직접 실행해야 하는 특수 연구 환경이다.
- 자유도 해석: 사용자가 인스턴스 전체에 대한 루트 액세스 권한을 가지며, OS 운영 및 보안에 대한 모든 책임을 진다는 의미이다.
### Container (표준화 및 이식성)
- 적합 상황: 마이크로서비스 아키텍처, 개발/운영 환경 간의 일관성(이식성)이 중요한 워크로드, 복잡하지만 장기 실행이 필요한 백엔드 서비스이다.
- Fargate의 역할: Fargate는 컨테이너 워크로드에 서버리스의 이점(인프라 관리 제거, 유휴 비용 최소화)을 결합하여, 컨테이너 환경의 운영 복잡도를 극적으로 낮춘다.
### Lambda (이벤트 중심 및 비용 최적화)
- 적합 상황: 데이터 변환, API 게이트웨이를 통한 짧은 백엔드 요청 처리, S3 업로드 등 이벤트에 반응하여 실행되는 간헐적이고 짧은 작업이다.
- 서버리스의 핵심 이점: 용량 예측 및 계획이 완전히 불필요하며, 트래픽이 0일 때 비용이 0이므로 비용 최적화 측면에서 가장 유리하다.
## Fargate vs. Lambda 상세 비교
- 두 서비스 모두 AWS에서 정의하는 서버리스 모델에 속한다.
- Lambda는 단일 함수의 실행에 초점을 맞추며, 실행 시간이 15분으로 제한되어 짧은 작업에 최적화된다.
- Fargate는 전체 애플리케이션(컨테이너)의 실행에 초점을 맞추며, 실행 시간 제한이 길어 지속적인 서비스 (웹 서버, API 백엔드)에 최적화된다.
## 비용 최적화 선택 기준
- 지속적인 로드 & 높은 트래픽: 컨테이너(Fargate 또는 ECS/EKS)를 사용하여 Scale-out하고 비용을 최적화한다.
- 간헐적인 로드 & 짧은 실행 시간: Lambda를 사용하여 요청/실행 시간당 비용을 지불한다.
- 특정 OS/커널 요구 사항: EC2를 사용하고, ASG 및 예약 인스턴스(Reserved Instance) 등을 통해 비용을 관리한다.
