# AWS Container Services (ECS, EKS, Fargate)
## 정의
애플리케이션과 모든 종속 항목을 경량의 표준화된 환경에 패키징하여 배포 및 실행하며, 이를 자동화된 방식으로 관리하는 AWS의 컨테이너 오케스트레이션 서비스
## 요소
### Container Image (컨테이너 이미지)
애플리케이션 실행에 필요한 코드, 런타임, 시스템 도구, 라이브러리 및 설정을 포함하는 불변의(Immutable) 패키지이다.
### Docker
컨테이너 이미지를 생성하고 실행하기 위한 사실상의 표준 기술이다.
### Amazon ECS (Elastic Container Service)
AWS에 최적화된 관리형 컨테이너 오케스트레이션 서비스이다. AWS 고유의 API를 사용하여 컨테이너화된 애플리케이션을 손쉽게 배포, 관리 및 확장한다.
### Amazon EKS (Elastic Kubernetes Service)
오픈 소스 [[Kubernetes]]를 AWS에서 쉽게 실행할 수 있도록 관리해주는 서비스이다. Kubernetes 생태계의 도구와 표준을 활용하고자 할 때 사용한다.
### AWS Fargate
컨테이너를 실행하기 위한 서버(EC2 인스턴스)를 프로비저닝하거나 관리할 필요가 없는 서버리스 컴퓨팅 엔진이다. ECS와 EKS 모두 Fargate를 컴퓨팅 옵션으로 사용할 수 있다.
## 원리
### 이미지 기반 배포 (Image-based Deployment)
애플리케이션 코드를 포함한 컨테이너 이미지를 생성하고, 이 이미지를 컨테이너 레지스트리(예: ECR)에 저장한다.
ECS 또는 EKS는 해당 이미지를 가져와 컨테이너를 실행하고, 로드 밸런서와 통합하여 트래픽을 분산하는 방식으로 작동한다.
### 오버헤드 최소화 (Minimal Overhead)
컨테이너는 OS 커널을 호스트와 공유하므로, EC2와 같은 전통적인 가상 머신(VM) 대비 오버헤드가 적고 시작 시간이 빠르다.
## 특징
### Decoupling (느슨한 결합)
마이크로서비스 아키텍처에서 각 서비스(예: 사용자 인증, 결제 처리)를 개별 컨테이너로 분리하여 배포한다. 이는 서비스 간 장애 독립성을 높이고 독립적인 크기 조정을 가능하게 한다.
### Portability (이식성)
개발 환경, 테스트 환경, 프로덕션 환경 등 모든 환경에서 동일한 컨테이너 이미지를 사용하므로, 환경 간의 불일치로 인한 오류를 방지한다.
### 서버리스 옵션 (Fargate)
Fargate를 사용하면 컨테이너 실행에 필요한 용량을 요청 시 자동으로 프로비저닝하고, 실행 시간에 대해서만 비용을 지불하여 운영 복잡도와 유휴 비용을 절감한다.
## 비교
| 구분 | Container (ECS/EKS) | Virtual Machine (EC2) |
| :--- | :--- | :--- |
| 가상화 레벨 | OS 커널 공유 (운영 체제 가상화) | 하드웨어 가상화 |
| 오버헤드 | 낮음 (경량) | 높음 (각 VM마다 전체 OS 설치) |
| 시작 시간 | 초 단위 (빠름) | 분 단위 (느림) |
| 운영 책임 | 오케스트레이터 및 컨테이너 내부만 관리 | OS 패치, 보안, 인스턴스 전체 관리 (높음) |
| Scaling 단위 | 애플리케이션 서비스 단위로 확장 | VM 인스턴스 단위로 확장 |
## 예시
### 마이크로서비스 아키텍처 구축
1. 웹 서비스의 3가지 구성 요소(프런트엔드, 인증 서비스, 백엔드 API)를 각각 별도의 컨테이너 이미지로 만든다.
2. 이 컨테이너들을 Amazon ECS에 AWS Fargate 시작 유형으로 배포한다.
3. 인증 서비스에 트래픽이 집중될 경우, ECS Auto Scaling을 통해 인증 서비스 컨테이너만 수평적으로 확장하고, 나머지 서비스는 영향을 받지 않는다.
4. 개발팀은 OS 패치나 EC2 서버 관리에 신경 쓸 필요 없이 코드 개발 및 배포에만 집중한다.
