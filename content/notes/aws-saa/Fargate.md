# AWS Fargate
## 정의
AWS의 서버리스 컴퓨팅 엔진으로, 컨테이너를 실행하기 위해 기본 인프라(가상 머신, OS, 패치)를 직접 프로비저닝하거나 관리할 필요가 없도록 해주는 기술
## 원리
### 서버리스 컨테이너 실행 환경
- Fargate는 [[ECS]] 또는 [[EKS]] 클러스터에서 컨테이너 작업(Task)을 실행할 때 선택할 수 있는 런치 타입(Launch Type) 중 하나이다.
- 사용자는 컨테이너 실행에 필요한 CPU 및 메모리 요구 사항만 정의하면, AWS가 해당 요구 사항을 충족하는 컴퓨팅 리소스를 자동으로 프로비저닝하고 관리한다.
### 인프라 관리의 추상화
- 기본 인스턴스의 OS 패치, 보안 강화(Hardening), 클러스터 스케일링 등의 모든 인프라 운영 책임을 AWS로 이전하여, 사용자는 애플리케이션 개발에만 집중할 수 있게 한다.
## 요소
- Fargate 런치 타입 (Launch Type): [[ECS]] 서비스 또는 작업을 정의할 때 인프라를 AWS가 관리하도록 지정하는 옵션이다.
- CPU/메모리 정의: 작업 정의(Task Definition) 내에서 Fargate가 프로비저닝할 최소 및 최대 CPU 및 메모리 조합을 명시적으로 정의해야 한다.
- Task: Fargate에서 실행되는 기본 컴퓨팅 단위(하나 이상의 컨테이너)이다. 각 Task는 자체적인 [[VPC]] 네트워크 인터페이스(ENI)를 얻는다.
## 특징
- 운영 간소화: 기본 호스트에 대한 SSH 접근, OS 업데이트, 클러스터 용량 계획 및 관리가 완전히 제거된다.
- 비용 효율성: 컨테이너가 실행되는 기간 동안의 CPU 및 메모리 리소스 사용량에 대해서만 비용을 지불한다. 사용하지 않는 인스턴스에 대한 비용이 발생하지 않는다.
- 확장성: 작업 정의와 서비스 설정을 기반으로, 트래픽 수요 변화에 따라 컨테이너 작업이 신속하고 탄력적으로 확장된다.
## 비교
| 구분 | Fargate (서버리스) | [[EC2]] (ECS EC2 타입) |
| :--- | :--- | :--- |
| 인프라 관리 주체 | AWS가 전적으로 담당 | 사용자가 VM, OS 패치 및 클러스터 용량 관리 |
| 비용 모델 | 작업 실행 시간 (CPU/메모리) | 프로비저닝된 [[EC2]] 인스턴스 시간 |
| 적합한 경우 | 개발 속도 우선, 관리 최소화, 예측 불가능한 워크로드 | 커스텀 OS나 특정 인스턴스 제어가 필요할 때 |
## 예시
- 웹 서비스 마이크로서비스 배포:
    1. 개발팀은 서비스의 Docker 이미지만 생성하여 [[ECR]]에 푸시한다.
    2. [[ECS]] 작업 정의에서 Fargate 런치 타입을 지정하고 CPU 0.5 vCPU, 메모리 1GB를 할당한다.
    3. ECS 서비스가 배포되면, Fargate가 자동으로 해당 리소스를 갖춘 환경에서 컨테이너를 실행하며, 개발자는 서버 관리 없이도 마이크로서비스를 운영할 수 있다.
