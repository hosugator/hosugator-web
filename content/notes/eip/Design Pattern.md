# 디자인 패턴 (Design Patterns)

## 정의

소프트웨어 설계 시 자주 발생하는 문제에 대한 검증된 솔루션

## Creational Patterns

객체 생성 방법을 캡슐화하여, 객체의 생성과 조합을 시스템으로부터 분리하는 패턴이다.

|           패턴 명칭           | 목적 및 특징                                                                 |
| :-----------------------: | :---------------------------------------------------------------------- |
| Factory Method (팩토리 메서드)  | 객체 생성을 위한 인터페이스를 정의하고, 실제로 생성할 클래스는 서브 클래스가 결정하도록 위임한다.                 |
| Abstract Factory (추상 팩토리) | 구체적인 클래스를 지정하지 않고 관련 객체들의 패밀리를 생성하기 위한 인터페이스를 제공한다.                 |
|       Builder (빌더)        | 복잡한 객체의 생성 단계를 분리하여 동일한 생성 절차에서 다양한 표현을 만들 수 있게 한다.                     |
|      Singleton (싱글턴)      | 클래스의 인스턴스가 오직 하나만 존재하도록 보장하고, 어디서든 그 인스턴스에 접근할 수 있는 전역적인 접근점을 제공한다. |
|     Prototype (프로토타입)     | 기존 객체를 복사(Clone)하여 새로운 객체를 생성함으로써 객체 생성 비용을 줄인다.                        |

## Structural Patterns

클래스나 객체들을 조합하여 더 크고 유연한 구조를 만드는 방법을 다룬다.

|         패턴 명칭          | 목적 및 특징                                                    |
| :--------------------: | :--------------------------------------------------------- |
|     Adapter (어댑터)      | 호환되지 않는 인터페이스를 가진 두 객체가 함께 작동할 수 있도록 변환해 준다. (일종의 '번역기')   |
|    Bridge (브리지)    | 추상화와 구현을 분리하여 독립적으로 변화할 수 있도록 연결한다.                        |
|    Composite (컴포지트)    | 객체들의 집합과 개별 객체를 동일하게 다룰 수 있도록 트리 구조를 구성한다.             |
| Decorator (데코레이터)  | 객체에 새로운 기능을 동적으로 추가하여 기능을 확장한다. (상속 대신 합성 사용)          |
|      Facade (퍼사드)      | 서브 시스템을 구성하는 복잡한 인터페이스 집합에 대해 단일하고 통합된 인터페이스를 제공한다.    |
| Flyweight (플라이웨이트) | 다수의 객체가 공유할 수 있는 경량(lightweight) 객체를 만들어 메모리 사용량을 줄인다.     |
|  Proxy ([[Proxy]])   | 어떤 객체에 대한 접근을 제어하기 위해 대리 객체를 둔다. (접근 제어, 지연 초기화 등에 사용) |
|                        |                                                            |

## Behavioral Patterns

객체 간의 상호작용과 책임 분배 방식을 다루며, 객체들이 어떻게 협력하는지 정의한다.

| 패턴 명칭 | 목적 및 특징 |
| :---: | :--- |
| Chain of Responsibility (책임 연쇄) | 요청을 처리할 기회를 한 객체에서 다른 객체로 넘겨주는 방식으로 처리 객체들을 연결한다. |
| Command (커맨드) | 요청을 객체로 캡슐화하여, 요청의 발신자와 수신자를 분리하고 요청을 매개변수화(저장, 로그, 취소)할 수 있게 한다. |
| Interpreter (인터프리터) | 언어의 문법을 정의하고, 그 문법에 따라 문장을 해석하는 방법을 제공한다. |
| Iterator (이터레이터) | 내부 구조를 노출하지 않고 컬렉션 객체의 요소들을 순차적으로 접근하는 방법을 제공한다. |
| Mediator (중재자) | 객체 간의 복잡한 상호작용을 캡슐화하는 중재자 객체를 두어 결합도를 낮춘다. |
| Memento (메멘토) | 객체의 내부 상태를 외부화하여, 객체 자체를 캡슐화하지 않고도 이전 상태로 되돌릴 수 있게 한다. |
| Observer (옵저버) | 객체 상태가 변할 때 의존하는 다른 객체들에게 알림을 보내 자동으로 갱신되도록 하는 일대다(1-to-N) 의존성을 정의한다. |
| State (상태) | 객체의 내부 상태에 따라 객체의 행위를 변경하게 한다. 마치 클래스가 바뀐 것처럼 보이게 한다. |
| Strategy (전략) | 알고리즘 패밀리를 정의하고, 각 알고리즘을 캡슐화하여 상호 교환 가능하게 한다. |
| Template Method (템플릿 메서드) | 알고리즘의 뼈대(템플릿)를 정의하고, 일부 단계를 서브 클래스가 구현하도록 위임한다. |
| Visitor (방문자) | 객체 구조에서 처리 기능을 분리하여, 새로운 기능을 쉽게 추가할 수 있게 한다. |
